# 接口废弃与删除执行方案

## 目标

清理旧研发方向遗留接口，逐步收敛到统一后端契约。

## 不可违反原则

- 先废弃、后删除
- 未经过观察窗口，不做物理删除
- 保留回滚开关直到稳定

## 判定标准（唯一依据）

统一契约是目标接口集；不在目标契约内的接口进入废弃候选。

当前第一批旧接口范围：

- `/v1/**`
- `/agents/**`

当前替代入口：

- `/catalog/resources`
- `/catalog/resources/{type}/{id}`
- `/catalog/resolve`
- `/invoke`

## 两阶段下线

### 第一阶段：废弃标记 + 禁写

- 在文档中标注 deprecated
- 在后端日志打废弃调用标记
- 旧接口写操作禁用（返回 `410 Gone`）
- 明确替代接口路径

当前已落地状态：

- `/v1/**`、`/agents/**` 已自动注入废弃响应头
- 上述路径的 `POST/PUT/PATCH/DELETE` 已禁写返回 `410`

### 第二阶段：物理删除

- 删除 Controller 路由
- 删除 Service 分支
- 删除仅服务旧接口的 DTO/Entity/Mapper 分支
- 删除相关测试与死代码

## 观察清单模板

每个待删接口都要记录：

- 接口路径
- 模块负责人
- 替代接口
- 最后一次调用时间
- 最近 7 天调用量
- 依赖系统阻塞项
- 删除审批人

## 回滚策略

- 保留旧接口开关，可临时恢复第一阶段行为
- 观察窗口内保持数据层兼容
- 若出现严重回归，回退到第一阶段，不直接回到旧实现
