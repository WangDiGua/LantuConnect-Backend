# SQL Migrations Guide

本目录用于存放**增量数据库迁移脚本**。  
目标：保证多人协作时迁移脚本可追踪、可重复执行、可审计。

## 1. 命名规范

统一使用如下格式：

`YYYYMMDD_NN_<action>_<target>.sql`

示例：

- `20260324_01_safe_add_resource_version.sql`
- `20260324_02_add_alert_indexes.sql`
- `20260325_01_fix_quota_unique_key.sql`

规则说明：

- `YYYYMMDD`：脚本创建日期（北京时间）。
- `NN`：当天顺序号（两位，从 `01` 开始）。
- `<action>`：动词短语，推荐：`add` / `alter` / `fix` / `backfill` / `drop`。
- `<target>`：受影响对象，推荐用表名或模块关键词。
- 仅使用小写字母、数字、下划线。

## 2. 脚本内容规范

每个脚本建议包含以下结构：

1. 头部注释（目的、作者、风险、是否幂等）。
2. 变更步骤（按执行顺序编号并加注释）。
3. 执行后校验 SQL（必须）。
4. 回滚说明（必须，注明是否破坏性）。

推荐原则：

- 默认写成**幂等脚本**（`IF NOT EXISTS`、条件插入等）。
- 默认写成**非破坏性脚本**（禁止直接 `DROP COLUMN` / `DROP TABLE`）。
- 破坏性操作必须在标题和注释中明确标注 `DANGEROUS`。

## 3. 执行顺序

按文件名**字典序**执行（也是时间序）。

建议流程：

1. 先在测试库执行。
2. 核对校验 SQL 输出。
3. 再在预发执行并验证应用功能。
4. 最后在生产低峰执行。

## 4. 协作流程（多人开发）

提交迁移脚本前请遵循：

1. 先 `pull` 最新代码。
2. 确认当天最新序号，避免重号。
3. 一个 PR 可包含多个脚本，但应按步骤拆分（不要把无关变更混在一个文件）。
4. PR 描述必须包含：
   - 影响表/索引
   - 是否幂等
   - 风险等级
   - 回滚方案

## 5. 回滚与应急

- 每个脚本必须提供回滚说明（可手动）。
- 如果回滚是破坏性的（如删表/删数据），必须在说明中显式标注并要求备份。
- 生产执行前，务必先做备份或快照。

## 6. 与基线 SQL 的关系

- `sql/lantu_connect.sql`：用于初始化全量基线。
- `sql/migrations/*.sql`：用于基线之后的增量演进。

不要把线上增量变更只改在基线文件而不提供 migration。
